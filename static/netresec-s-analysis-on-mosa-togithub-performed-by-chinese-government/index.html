以下是三月27日Github博客记载的内容：

我们正遭受本站历史上规模最大的分布式拒绝服务攻击。该次攻击开始于三月26日，周四下午2时，并集成大量不同方式与来源。此次攻击包含之前我站受到的所有攻击方式，与此同时添加较为复杂的新技术，可以利用大量无关网民的浏览器对github.com制造大量的通信数据量。根据目前收到的报告，基本可以判断本次攻击的意图是促使我们于本站移除特定的内容。

针对本次攻击的详细调查显示中国使用主动/被动网络设施实施了数据包注入攻击，即对github进行所谓的旁观者攻击。确认此次攻击属于旁观者攻击的调查结论来自于此篇博客底部的TTL分析数据。

此次中间人攻击简单来说由以下步骤实现：

  1. 来源：在中国境外的某个无辜使用者正在浏览互联网
  2. 而网页内容中存在一个保存在中国服务器的javasript，例如百度分析scipt,其常被网站管理员用于追踪访问者的信息(与google Analytics作用类似)
  3. 随后，当请求进入中国网域，中国被动网络架构将探测到浏览器对百度javasript的访问请求
  4. 来源于中国境内的虚假响应包取代了真实的baidu analyticsn script返回给浏览器。虚假响应包含恶意脚本，此脚本操纵用户浏览器反复加载Github上两个特定页面。

事实上，不是所有从中国境内加载脚本的用户都会中招。分析显示大约1%请求baidu analytics script的请求会收到恶意脚本

TTL分析

存活时间(TTL)验定是分析中间人攻击和旁观者攻击的最有力的工具。此前，本站曾应用此方法分析中国对iCloud，Yahoo，Google，Github发动的中间人攻击。

本次对Github的攻击中比较有趣的地方在于攻击者试图通过修改注入数据包IP ttl值隐藏恶意脚本的注入点。

以下数据显示Source-IP, Destination-IP, TCP-Flags and IP-TTL：

tshark -r baidu-high-ttl.pcap -T fields -e ip.src -e ip.dst -e tcp.flags -e ip.ttl 192.168.70.160 61.135.185.140 0x0002 64 <- SYN (client) 61.135.185.140 192.168.70.160 0x0012 42 <- SYN+ACK (server) 192.168.70.160 61.135.185.140 0x0010 64 <- ACK (client) 192.168.70.160 61.135.185.140 0x0018 64 <- HTTP GET (client) 61.135.185.140 192.168.70.160 0x0018 227 <- Injected packet 1 (injector) 192.168.70.160 61.135.185.140 0x0010 64 61.135.185.140 192.168.70.160 0x0018 228 <- Injected packet 2 (injector) 61.135.185.140 192.168.70.160 0x0019 229 <- Injected packet 3 (injector) 192.168.70.160 61.135.185.140 0x0010 64 192.168.70.160 61.135.185.140 0x0011 64

请注意服务器发送的syn+ack包ttl值为42，有效载荷内的三个注入数据包的ttl值分别为227,228和229,为什么会出现这种情况？

以下是另外一个PCAP文件，指示注入数据包的ttl 值较低：

tshark -r baidu-low-ttl.pcap -T fields -e ip.src -e ip.dst -e tcp.flags -e ip.ttl 192.168.70.160 61.135.185.140 0x0002 64 <- SYN (client) 61.135.185.140 192.168.70.160 0x0012 42 <- SYN+ACK (server) 192.168.70.160 61.135.185.140 0x0010 64 <- ACK (client) 192.168.70.160 61.135.185.140 0x0018 64 <- HTTP GET (client) 61.135.185.140 192.168.70.160 0x0018 30 <- Injected packet 1 (injector) 192.168.70.160 61.135.185.140 0x0010 64 61.135.185.140 192.168.70.160 0x0018 31 <- Injected packet 2 (injector) 61.135.185.140 192.168.70.160 0x0019 32 <- Injected packet 3 (injector) 192.168.70.160 61.135.185.140 0x0010 64 192.168.70.160 61.135.185.140 0x0011 64

服务器的syn+ack包在整个分析过程中ttl值保持在42左右，与此同时携带恶意脚本的有效载荷的ttl值在30与229之间浮动。上述行为指出：以上的syn+ack包确实由百度服务器发出，恶意脚本的注入地点不是服务器端，而是在其他地点。

如上所述，三个注入数据包携带的负载总是相同的，唯一不同的是每个会话的目标tcp端口。此结论有利的支持了这三个数据包来源于人为注入的判断。当实验故意丢弃其中一个数据包并请求服务器重新传送该数据包时，服务器并不会响应与重新传送。至此，恶意脚本取代实际的百度服务器端的js脚本进行响应的事实清晰可见，也揭示出本次攻击的注入包来源于旁观者攻击。

恶意脚本取代的其他正常脚本

Baidu analytics不是本次攻击中唯一被取代的脚本对象。用户报告中也有提及百度广告脚本以及其他一些百度服务脚本同样被恶意脚本所取代。GreatFire.org的技术报告中，列出了以下被攻击者替代的URL内容，进行分布式拒绝服务攻击的脚本：

  • hm.baidu.com/h.js
  • cbjs.baidu.com/js/o.js
  • dup.baidustatic.com/tpl/wh.js
  • dup.baidustatic.com/tpl/ac.js
  • dup.baidustatic.com/painter/clb/fixed7o.js
  • dup.baidustatic.com/painter/clb/fixed7o.js
  • eclick.baidu.com/fp.htm?br= …
  • pos.baidu.com/acom?adn= …
  • cpro.baidu.com/cpro/ui/uijs.php?tu=…
  • pos.baidu.com/sync_pos.htm?cproid=…

上述的域均属于百度公司所有，但技术层面上讲，任何中国国内网站的js脚本都可以被类似的方法劫持并实施注入攻击。

总结：

本次攻击揭示了中国被动/主动网络过滤设施规模之庞大，防火长城已经可以应用于实施强力的分布式拒绝服务攻击。因此，防火长城不应该被视为监察和审查中国网民网络流量的技术，而是一个可以借助访问中国网站用户之手施展DDOS攻击的网络武器。

更新－四月13日

Bill Marczak, Nicholas Weaver, Jakub Dalek, Roya Ensafi, David Fifield, Sarah McKune, Arn Rey, John Scott-Railton, Ronald Deibert and Vern Paxson 公布了他们关于所谓大炮的新式网络武器研究。在博客中，他们确认了本站关于注入数据包怪异ttl数据的发现。

所谓大炮注入的数据包与防火长城注入的数据包拥有相同且独特的ttl边信道，也就是暗示防火长城与大炮之间可能共享某些代码。

建议阅读Usenix FOCI的文章，其展示了防火长城DNS审查的宏观架构。

他分享了防火长城和大炮的源代码和相同位置，也揭示了大炮可能在防火长城原有的架构上研发形成。
